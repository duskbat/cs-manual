### TCP 三次握手

C->S 请求建立连接: SYN=1, seq = client_isn
S->C 连接请求确认: SYN=1, ack=client_isn + 1, seq = server_isn
C->S 确认建立连接: SYN=0, ack=server_isn + 1, seq = client_isn + 1

两次握手的问题: 如果两次握手就建立连接，如果客户端接收异常，服务端会认为已经建立了连接，等待通信；而客户端认为没有建立连接。

### TCP 四次挥手

C->S: FIN bit=1
S->C: ACK
S->C: FIN bit=1
C->S: ACK
TCP 半关闭，收发单向关闭

### TCP 的可靠性

TCP 发送方的三个主要事件:

1. 接收数据生成报文段交给 IP
2. 定时器超时重传
3. 根据返回的 ACK 确认 SendBase

## 一个完整的 web 请求

```c
输入URL
-> DNS解析: 浏览器缓存->路由器缓存->DNS 域名解析
-> 建立 TCP连接 -> 发送 HTTP请求
```

### Cookie Session

cookie 和 session 都是用来**跟踪浏览器用户身份**的会话方式。

> 不要太注重下面乱七八糟的描述

cookie 创建过程:

1. 浏览器端第一次发送请求到服务器端
2. 服务器端创建 Cookie，该 Cookie 中包含用户的信息，然后将该 Cookie 发送到浏览器端
3. 浏览器端再次访问服务器端时会携带服务器端创建的 Cookie
4. 服务器端通过 Cookie 中携带的数据区分不同的用户

session 创建过程:

1. 浏览器端第一次发送请求到服务器端，服务器端创建一个 Session，同时会创建一个特殊的 Cookie（name 为 SESSIONID 的固定值，value 为 session 对象的 ID），然后将该 Cookie 发送至浏览器端
2. 浏览器端发送第 N（N>1）次请求到服务器端, 浏览器端访问服务器端时就会携带该 name 为 JSESSIONID 的 Cookie 对象
3. 服务器端根据 Cookie 里 name 为 SESSIONID 的 value(sessionId),去查询 Session 对象，从而区分不同用户。
   - name 为 JSESSIONID 的 Cookie 不存在（**关闭或更换浏览器**），返回 1 中重新去创建 Session 与特殊的 Cookie
   - name 为 JSESSIONID 的 Cookie 存在，根据 value 中的 SessionId 去寻找 session 对象
     - value 为 SessionId 不存在（**Session 对象默认存活 30 分钟**），返回 1 中重新去创建 Session 与特殊的 Cookie
     - value 为 SessionId 存在，返回 session 对象

区别:
cookie 只能存储 ASCII 码字符串
cookie 数据保存在客户端，session 数据保存在服务端。
cookie 不安全, 可以 mock

## HTTP

### HTTP 状态码

1XX 信息性状态码
2XX 成功状态码
3XX 重定向状态码

- 301 永久重定向
- 302 暂时重定向

4XX 客户端错误状态码 服务器无法处理请求

- 400 Bad Request
- 401 Unauthorized
- 403 Forbidden
- 404 Not Found 请求失败，所请求的资源未在服务器上发现

5XX 服务器错误状态码 服务器处理请求出错

- 500 Internal Server Error 内部错误
- 502 Bad Gateway 网关收到上游的错误相应
- 503 Service Unavailable 服务器维护或者过载

### 对比 HTTPS

端口 80 / 443
使用了隧道加密通信，http 先和 ssl 通信，ssl 再和 tcp 通信。

> SSL 跟 TLS 区别不大, SSL 从 3.0 才发布, 升级后版本就是 TLS1.0 了

https 使用了混合加密，使用非对称加密(证书验证)来加密对称加密密钥，然后使用对称加密进行通讯。

> 公钥私钥的生成依赖于单向陷门函数

#### 整体过程

**证书验证:**
浏览器发起 HTTPS 请求
服务端返回证书
客户端验证证书

**数据传输**
根据证书生成随机数
通过证书里面的公钥加密随机数密钥, 发送给服务器
服务器通过私钥解密出密钥
服务器通过密钥对称加密传输内容

解决中间人攻击的办法:
认证：
数字证书认证机构（CA，Certificate Authority）是客户端与服务器双方都可信赖的第三方机构。分配公钥匙，绑定证书。

1. CA 持有 S 公钥, CA 证书 hash 成摘要并用 CA 私钥加密;
2. S 获得 CA 证书和加密后的摘要;
3. C 持有 CA 公钥, S 将证书传给 C, 可以解密证书上的签名, 并 hash 处理证书, 对比证书是否合法;
4. 合法则信任 S 的公钥.

## OSI 7 层模型

- 7 应用层: 为计算机用户提供服务
- 6 表示层: 数据处理(编解码 加密解密 压缩解压)
- 5 会话层: 管理(建立 维护 重连)应用程序之间的会话
- 4 传输层: 为两台主机进程之间的通信提供的数据传输服务
- 3 网络层: 控制子网的运行
- 2 数据链路层: 物理寻址, 比特流转换为逻辑传输线路
- 1 物理层: 比特流传输

## TCP/IP 4 层模型

- 4 应用层: 应用 表示 会话 定义了信息交换格式
- 3 传输层: 传输 TCP 就是传输层协议
- 2 网络层: 网络 IP 协议就是网络层协议 传输层的报文段被封装成包
- 1 网络接口层: 数据链路 物理

### 对称加密非对称加密
