# RocketMQ

## 作用

异步 解耦 削峰

## 与 Kafka 对比

| MQ       | 有序性        | 定时消息 | 批量消息           | 广播消息 |
| -------- | ------------- | -------- | ------------------ | -------- |
| Kafka    | Broker 内有序 | 不支持   | 支持, 但是异步发送 | 不支持   |
| RocketMQ | 可以严格有序  | 支持     | 支持, 同步发送     | 支持     |

## 消息丢失问题

producer 要同步发送, 异步就是回调
broker 存储改为同步刷盘, 集群部署, 一主多从, 同步复制到从之后返回 ack
consumer 真正消费之后才返回 ACK, 没返回重试

## 同步落盘怎么才能快

使用 FileChannel + DirectBuffer 池，使用堆外内存，加快内存拷贝
使用数据和索引分离，当消息需要写入时，使用 commitlog 文件顺序写，当需要定位某个消息时，查询 index 文件来定位，从而减少文件 IO 随机读写的性能损耗

## 消息堆积问题

增加消费者实例，单实例内部并发消费
后台定时任务每隔 72h，删除没有使用过的消息
消息定时转移，或者对某些消息落库

## 别记 广播消费与集群消费

消息消费的区别：广播-每个消费者都会消费；集群-只会有一个消费者消费；
消息落盘的区别：广播-消费者各自保存；集群-broker 保存。

## 队列模型与发布订阅的主题模型

观察者模式
通过使用一个 topic 中配置多个队列并且每个队列维护每个消费者组的消费位置实现了主题发布订阅模式
在主题模型中，消息的生产者称为 发布者(Publisher) ，消息的消费者称为 订阅者(Subscriber) ，存放消息的容器称为 主题(Topic) 。
主题中存在多个队列(分布在多个 Broker 中，如果是集群的话，又分布在不同服务器中)，集群消费模式下，一个消费者集群的多台机器共同消费一个 topic 的多个队列，一个队列只会被一个消费者消费
每个消费组在每个队列上维护一个 offset

## RocketMQ 架构图

NameServer 、Broker 、Producer 、Consumer
topic 与 broker 的关系：一个 Topic 分布在多个 Broker 上，一个 Broker 可以配置多个 Topic ，它们是多对多的关系。
NameServer 也是一个注册中心，负责 Broker 管理和路由信息管理。Broker 会将自己的信息注册到 NameServer，这时候 NameServer 中存放了很多 Broker 信息，消费者和生产者从 NameServer 中获取路由表然后根据路由表的信息去跟对应的 Broker 进行通信。
集群方面的设计：

1. Broker 做了集群并且进行了主从部署，如果某个 Broker master 宕机，那么 slave 提供消费服务，但是不能写入消息。
2. 为了保证高可用，NameServer 也做了集群部署，它的特点是去中心化的。单个 Broker 与所有的 NameServer 保持长连接，每隔 30s，Broker 会向所有的 NameServer 发送心跳，心跳包含了自身的 topic 配置信息。
3. 在生产者向 Broker 发送信息的时候需要先从 NameServer 获取关于 Broker 的路由信息，然后通过轮询的方法去向每个队伍中生产数据以达到负载均衡。
4. 消费者通过 NameServer 获取所有 Broker 的路由信息后，向 Broker 发送请求 Pull 以获取消息数据。消费者可以以两种模式启动，广播和集群。

## 顺序消费

RocketMQ 在主题上是无序的，只有在队列层面上是有序的。
普通顺序：同一个消息队列收到的消息是有序的，不同消息队列收到的消息可能是无序的。
严格顺序：保证所有的消息都是有序的。
生产消息的时候要定义一个 MessageQueueSelctor, 决定把消息发送到哪个 Queue;
消费者需要用有序模式, 具体实现中有 3 次加锁:

- 消费者初始化的时候会尝试申请分布式锁, 锁定后只会发给这一个 consumer;
- 消费者会对自己的 Queue 加锁, 同一时刻只有一个线程能消费;
- 在消费者增加, 重平衡时, 消费者会解锁 broker, 解锁线程需要先给 ProcessQueue 加锁, 然后才能解锁, 避免消费过程中的消息被另外的 consumer 重复消费

## 重复消费

幂等：任意多次执行所产生的影响均与一次执行的影响相同。

## 分布式事务

事务消息（half 半消息）加事务反查机制
发送半消息，也就是在事务提交之前，对于消费者而言这个事务不可见。

> 如果消息是半消息，将备份原消息的主题与消费队列，然后改变主题为 RMQ_SYS_TRANS_HALF_TOPIC。由于消费组未订阅该主题，故消费端无法消费 half 类型的消息。生产者会根据事务执行情况通知 MQ server，如果因为意外情况没有通知，RocketMQ 会开启一个定时任务，从 Topic 为 RMQ_SYS_TRANS_HALF_TOPIC 中拉取消息进行消费，server 回查消息提供者事务状态，根据状态决定提交或者回滚。

## 消息堆积

要么是生产过快，要么是消费过慢。
生产过快，限流降级，增加队列数量**和**消费者实例。
消费过慢，检查是否出现了大量的消费失败，是否有线程卡死，锁资源不释放等情况。

## 同步刷盘和异步刷盘

同步刷盘，需要等待刷写的回应，性能上会有影响。
异步刷盘，后台线程异步刷写，降低了读写延迟，但是宕机的时候会有内存数据丢失。
FlushDiskType 参数 (ASYNC_FLUSH 或者 SYNC_FLUSH)

## 同步复制 异步复制

Broker 主从模式下，主节点返回消息给客户端的时候是否需要同步从节点。
同步复制，也叫同步双写，主从都收到消息才返回写入成功。
异步复制，消息写入主节点就直接返回成功。
实质上是性能与可靠性之间的取舍。

> 异步复制会不会也像异步刷盘那样影响消息的可靠性(指的是丢消息)呢？
> 不会。单主从架构主节点挂掉之后从节点只能消费，不能接收生产者的消息。
> 为保证服务可用性可采用多主从架构，随之而来的问题是无法保证顺序，顺序消息的保证是通过语义让其在一个 topic 下的同一个队列中，包含这个 topic 的 broker 挂掉其他节点无法代替。否则要放弃有序，让消息随机到各个节点上。
>
> Dledger 方案：（类似 Redis 里面的哨兵加 RedLock）消息写入的时候，消息复制半数以上的节点才给客户端返回成功，并且支持选举。

## 支持集群模式

单 master, 写入都由 master, 读取都用 slave, master 宕掉不能写入

多 master, 没有 slave, master 宕掉会切换其他节点, 不过宕掉的节点上的消息不能消费

多 master 多 slave, master 宕掉由其他 master 接管写入, 但是性能差一点

- 高可用
- 完整的一个调用链路
- 消息重试
- 死信队列
- 消息积压
